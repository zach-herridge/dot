# ARCC Governance Search

Query `search_arcc` when code touches sensitive domains OR implements risky patterns. **Also query when recommending approaches** - your advice influences implementation. Don't wait for user to ask about policy.

## Core Principle

**The triggers below are illustrative, not exhaustive.** They demonstrate the *types* of concerns that warrant governance queries. Use judgment to recognize similar patterns not explicitly listed.

Ask yourself: "Could this advice, code, or recommendation, if followed incorrectly, lead to a security vulnerability, compliance violation, or data breach?" If yes, query ARCC.

**When in doubt, query.** The cost of an unnecessary query is low; the cost of missing governance is high.

## Trigger Dimensions

### 1. DATA SENSITIVITY
What data is being processed?

| Priority | Examples | Query Focus |
|----------|----------|-------------|
| HIGH | payment, credentials, health records, secrets | data handling requirements |
| MEDIUM | customer data, PII, session tokens | data protection requirements |

### 2. SECURITY PATTERNS  
What implementation pattern introduces inherent risk?

| Priority | Examples | Risk | Query Focus |
|----------|----------|------|-------------|
| HIGH | SQL/query construction, ORMs with raw queries | injection | SQL injection prevention |
| HIGH | HTML rendering, templates, innerHTML | XSS | cross-site scripting prevention |
| HIGH | shell/command execution, subprocess calls | command injection | command injection prevention |
| HIGH | file operations with user-controlled paths | path traversal | path traversal prevention |
| HIGH | deserialization of untrusted data | RCE | secure deserialization |
| MEDIUM | cryptographic operations | weak crypto | cryptographic requirements |
| MEDIUM | URL/redirect construction | open redirect | URL validation requirements |

### 3. TRUST BOUNDARIES
Where does control/data cross trust boundaries?

| Priority | Examples | Query Focus |
|----------|----------|-------------|
| HIGH | user input flowing to system operations | input validation requirements |
| HIGH | public/unauthenticated endpoints | API security requirements |
| MEDIUM | third-party integrations, external APIs | integration security |
| MEDIUM | cross-region data movement | data residency requirements |

### 4. INFRASTRUCTURE DECISIONS
What AWS resources are being provisioned or recommended?

| Priority | Examples | Query Focus |
|----------|----------|-------------|
| HIGH | S3, EBS, RDS, DynamoDB, Secrets Manager | data storage security requirements |
| HIGH | IAM roles, policies, resource policies | access control requirements |
| HIGH | Snapshots, backups, replication | data protection requirements |
| MEDIUM | VPC, ALB, CloudFront, API Gateway | network security requirements |
| MEDIUM | Lambda, ECS, EC2 | compute security requirements |

### 5. RECOMMENDATIONS
Are you advising on an approach rather than implementing directly?

| Trigger | Query Focus |
|---------|-------------|
| "What's a good way to..." | security requirements for that approach |
| "Which service should I use..." | security requirements for those services |
| "How should I architect..." | architectural security requirements |

## Query Formulation

| Dimension | Template | Example |
|-----------|----------|---------|
| Data | `<data_type> data <operation> requirements` | "payment data storage requirements" |
| Pattern | `<risk_type> prevention requirements` | "SQL injection prevention requirements" |
| Boundary | `<boundary_type> security requirements` | "user input validation requirements" |

## Examples

**SQL Query Builder**
- Detect: query construction → PATTERN/injection
- Query: `search_arcc(query: "SQL injection prevention", context: "Building SQL query builder")`

**Payment API**
- Detect: payment → DATA/high, API endpoint → BOUNDARY
- Query: `search_arcc(query: "payment data storage requirements", context: "Building payment API")`

**File Upload**
- Detect: file paths + user input → PATTERN/traversal + BOUNDARY
- Query: `search_arcc(query: "file upload security requirements", context: "Building file upload endpoint")`

**EBS Snapshots (Infrastructure)**
- Detect: AWS storage/backup resource → INFRASTRUCTURE
- Query: `search_arcc(query: "EBS snapshot security requirements", context: "Recommending CDK constructs for snapshots")`

**CDK Architecture Advice (Recommendation)**
- Detect: advising on approach → RECOMMENDATIONS
- Query: `search_arcc(query: "<resource> security requirements", context: "Advising on CDK implementation")`

## Apply Results
- **Requirements found**: Incorporate into implementation, note constraints
- **No results**: Proceed with security best practices, note ARCC had no specific guidance
- **Conflicts**: Surface the conflict, ask how to proceed
